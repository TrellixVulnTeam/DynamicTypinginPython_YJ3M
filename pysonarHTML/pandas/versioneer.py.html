  18 This is a tool for managing a recorded version number in distutils-based
  19 python projects. The goal is to remove the tedious and error-prone "update
  20 the embedded version string" step from your release process. Making a new
  21 release should be as easy as recording a new tag in your version-control
  22 system, and maybe making new tarballs.
  23 
  24 
  25 ## Quick Install
  26 
  27 * `pip install versioneer` to somewhere to your $PATH
  28 * add a `[versioneer]` section to your setup.cfg (see below)
  29 * run `versioneer install` in your source tree, commit the results
  30 
  31 ## Version Identifiers
  32 
  33 Source trees come from a variety of places:
  34 
  35 * a version-control system checkout (mostly used by developers)
  36 * a nightly tarball, produced by build automation
  37 * a snapshot tarball, produced by a web-based VCS browser, like github's
  38   "tarball from tag" feature
  39 * a release tarball, produced by "setup.py sdist", distributed through PyPI
  40 
  41 Within each source tree, the version identifier (either a string or a number,
  42 this tool is format-agnostic) can come from a variety of places:
  43 
  44 * ask the VCS tool itself, e.g. "git describe" (for checkouts), which knows
  45   about recent "tags" and an absolute revision-id
  46 * the name of the directory into which the tarball was unpacked
  47 * an expanded VCS keyword ($Id$, etc)
  48 * a `_version.py` created by some earlier build step
  49 
  50 For released software, the version identifier is closely related to a VCS
  51 tag. Some projects use tag names that include more than just the version
  52 string (e.g. "myproject-1.2" instead of just "1.2"), in which case the tool
  53 needs to strip the tag prefix to extract the version identifier. For
  54 unreleased software (between tags), the version identifier should provide
  55 enough information to help developers recreate the same tree, while also
  56 giving them an idea of roughly how old the tree is (after version 1.2, before
  57 version 1.3). Many VCS systems can report a description that captures this,
  58 for example `git describe --tags --dirty --always` reports things like
  59 "0.7-1-g574ab98-dirty" to indicate that the checkout is one revision past the
  60 0.7 tag, has a unique revision id of "574ab98", and is "dirty" (it has
  61 uncommitted changes.
  62 
  63 The version identifier is used for multiple purposes:
  64 
  65 * to allow the module to self-identify its version: `myproject.__version__`
  66 * to choose a name and prefix for a 'setup.py sdist' tarball
  67 
  68 ## Theory of Operation
  69 
  70 Versioneer works by adding a special `_version.py` file into your source
  71 tree, where your `__init__.py` can import it. This `_version.py` knows how to
  72 dynamically ask the VCS tool for version information at import time.
  73 
  74 `_version.py` also contains `$Revision$` markers, and the installation
  75 process marks `_version.py` to have this marker rewritten with a tag name
  76 during the `git archive` command. As a result, generated tarballs will
  77 contain enough information to get the proper version.
  78 
  79 To allow `setup.py` to compute a version too, a `versioneer.py` is added to
  80 the top level of your source tree, next to `setup.py` and the `setup.cfg`
  81 that configures it. This overrides several distutils/setuptools commands to
  82 compute the version when invoked, and changes `setup.py build` and `setup.py
  83 sdist` to replace `_version.py` with a small static file that contains just
  84 the generated version data.
  85 
  86 ## Installation
  87 
  88 First, decide on values for the following configuration variables:
  89 
  90 * `VCS`: the version control system you use. Currently accepts "git".
  91 
  92 * `style`: the style of version string to be produced. See "Styles" below for
  93   details. Defaults to "pep440", which looks like
  94   `TAG[+DISTANCE.gSHORTHASH[.dirty]]`.
  95 
  96 * `versionfile_source`:
  97 
  98   A project-relative pathname into which the generated version strings should
  99   be written. This is usually a `_version.py` next to your project's main
 100   `__init__.py` file, so it can be imported at runtime. If your project uses
 101   `src/myproject/__init__.py`, this should be `src/myproject/_version.py`.
 102   This file should be checked in to your VCS as usual: the copy created below
 103   by `setup.py setup_versioneer` will include code that parses expanded VCS
 104   keywords in generated tarballs. The 'build' and 'sdist' commands will
 105   replace it with a copy that has just the calculated version string.
 106 
 107   This must be set even if your project does not have any modules (and will
 108   therefore never import `_version.py`), since "setup.py sdist" -based trees
 109   still need somewhere to record the pre-calculated version strings. Anywhere
 110   in the source tree should do. If there is a `__init__.py` next to your
 111   `_version.py`, the `setup.py setup_versioneer` command (described below)
 112   will append some `__version__`-setting assignments, if they aren't already
 113   present.
 114 
 115 * `versionfile_build`:
 116 
 117   Like `versionfile_source`, but relative to the build directory instead of
 118   the source directory. These will differ when your setup.py uses
 119   'package_dir='. If you have `package_dir={'myproject': 'src/myproject'}`,
 120   then you will probably have `versionfile_build='myproject/_version.py'` and
 121   `versionfile_source='src/myproject/_version.py'`.
 122 
 123   If this is set to None, then `setup.py build` will not attempt to rewrite
 124   any `_version.py` in the built tree. If your project does not have any
 125   libraries (e.g. if it only builds a script), then you should use
 126   `versionfile_build = None` and override `distutils.command.build_scripts`
 127   to explicitly insert a copy of `versioneer.get_version()` into your
 128   generated script.
 129 
 130 * `tag_prefix`:
 131 
 132   a string, like 'PROJECTNAME-', which appears at the start of all VCS tags.
 133   If your tags look like 'myproject-1.2.0', then you should use
 134   tag_prefix='myproject-'. If you use unprefixed tags like '1.2.0', this
 135   should be an empty string.
 136 
 137 * `parentdir_prefix`:
 138 
 139   a optional string, frequently the same as tag_prefix, which appears at the
 140   start of all unpacked tarball filenames. If your tarball unpacks into
 141   'myproject-1.2.0', this should be 'myproject-'. To disable this feature,
 142   just omit the field from your `setup.cfg`.
 143 
 144 This tool provides one script, named `versioneer`. That script has one mode,
 145 "install", which writes a copy of `versioneer.py` into the current directory
 146 and runs `versioneer.py setup` to finish the installation.
 147 
 148 To versioneer-enable your project:
 149 
 150 * 1: Modify your `setup.cfg`, adding a section named `[versioneer]` and
 151   populating it with the configuration values you decided earlier (note that
 152   the option names are not case-sensitive):
 153 
 154   ````
 155   [versioneer]
 156   VCS = git
 157   style = pep440
 158   versionfile_source = src/myproject/_version.py
 159   versionfile_build = myproject/_version.py
 160   tag_prefix = ""
 161   parentdir_prefix = myproject-
 162   ````
 163 
 164 * 2: Run `versioneer install`. This will do the following:
 165 
 166   * copy `versioneer.py` into the top of your source tree
 167   * create `_version.py` in the right place (`versionfile_source`)
 168   * modify your `__init__.py` (if one exists next to `_version.py`) to define
 169     `__version__` (by calling a function from `_version.py`)
 170   * modify your `MANIFEST.in` to include both `versioneer.py` and the
 171     generated `_version.py` in sdist tarballs
 172 
 173   `versioneer install` will complain about any problems it finds with your
 174   `setup.py` or `setup.cfg`. Run it multiple times until you have fixed all
 175   the problems.
 176 
 177 * 3: add a `import versioneer` to your setup.py, and add the following
 178   arguments to the setup() call:
 179 
 180         version=versioneer.get_version(),
 181         cmdclass=versioneer.get_cmdclass(),
 182 
 183 * 4: commit these changes to your VCS. To make sure you won't forget,
 184   `versioneer install` will mark everything it touched for addition using
 185   `git add`. Don't forget to add `setup.py` and `setup.cfg` too.

 187 ## Post-Installation Usage
 188 
 189 Once established, all uses of your tree from a VCS checkout should get the
 190 current version string. All generated tarballs should include an embedded
 191 version string (so users who unpack them will not need a VCS tool installed).
 192 
 193 If you distribute your project through PyPI, then the release process should
 194 boil down to two steps:
 195 
 196 * 1: git tag 1.0
 197 * 2: python setup.py register sdist upload
 198 
 199 If you distribute it through github (i.e. users use github to generate
 200 tarballs with `git archive`), the process is:
 201 
 202 * 1: git tag 1.0
 203 * 2: git push; git push --tags
 204 
 205 Versioneer will report "0+untagged.NUMCOMMITS.gHASH" until your tree has at
 206 least one tag in its history.

 208 ## Version-String Flavors
 209 
 210 Code which uses Versioneer can learn about its version string at runtime by
 211 importing `_version` from your main `__init__.py` file and running the
 212 `get_versions()` function. From the "outside" (e.g. in `setup.py`), you can
 213 import the top-level `versioneer.py` and run `get_versions()`.
 214 
 215 Both functions return a dictionary with different flavors of version
 216 information:
 217 
 218 * `['version']`: A condensed version string, rendered using the selected
 219   style. This is the most commonly used value for the project's version
 220   string. The default "pep440" style yields strings like `0.11`,
 221   `0.11+2.g1076c97`, or `0.11+2.g1076c97.dirty`. See the "Styles" section
 222   below for alternative styles.
 223 
 224 * `['full-revisionid']`: detailed revision identifier. For Git, this is the
 225   full SHA1 commit id, e.g. "1076c978a8d3cfc70f408fe5974aa6c092c949ac".
 226 
 227 * `['dirty']`: a boolean, True if the tree has uncommitted changes. Note that
 228   this is only accurate if run in a VCS checkout, otherwise it is likely to
 229   be False or None
 230 
 231 * `['error']`: if the version string could not be computed, this will be set
 232   to a string describing the problem, otherwise it will be None. It may be
 233   useful to throw an exception in setup.py if this is set, to avoid e.g.
 234   creating tarballs with a version string of "unknown".
 235 
 236 Some variants are more useful than others. Including `full-revisionid` in a
 237 bug report should allow developers to reconstruct the exact code being tested
 238 (or indicate the presence of local changes that should be shared with the
 239 developers). `version` is suitable for display in an "about" box or a CLI
 240 `--version` output: it can be easily compared against release notes and lists
 241 of bugs fixed in various releases.
 242 
 243 The installer adds the following text to your `__init__.py` to place a basic
 244 version in `YOURPROJECT.__version__`:
 245 
 246     from ._version import get_versions
 247     __version__ = get_versions()['version']
 248     del get_versions
 249 
 250 ## Styles
 251 
 252 The setup.cfg `style=` configuration controls how the VCS information is
 253 rendered into a version string.
 254 
 255 The default style, "pep440", produces a PEP440-compliant string, equal to the
 256 un-prefixed tag name for actual releases, and containing an additional "local
 257 version" section with more detail for in-between builds. For Git, this is
 258 TAG[+DISTANCE.gHEX[.dirty]] , using information from `git describe --tags
 259 --dirty --always`. For example "0.11+2.g1076c97.dirty" indicates that the
 260 tree is like the "1076c97" commit but has uncommitted changes (".dirty"), and
 261 that this commit is two revisions ("+2") beyond the "0.11" tag. For released
 262 software (exactly equal to a known tag), the identifier will only contain the
 263 stripped tag, e.g. "0.11".
 264 
 265 Other styles are available. See details.md in the Versioneer source tree for
 266 descriptions.
 267 
 268 ## Debugging
 269 
 270 Versioneer tries to avoid fatal errors: if something goes wrong, it will tend
 271 to return a version of "0+unknown". To investigate the problem, run `setup.py
 272 version`, which will run the version-lookup code in a verbose mode, and will
 273 display the full contents of `get_versions()` (including the `error` string,
 274 which may help identify what went wrong).

 276 ## Updating Versioneer
 277 
 278 To upgrade your project to a new release of Versioneer, do the following:
 279 
 280 * install the new Versioneer (`pip install -U versioneer` or equivalent)
 281 * edit `setup.cfg`, if necessary, to include any new configuration settings
 282   indicated by the release notes
 283 * re-run `versioneer install` in your source tree, to replace
 284   `SRC/_version.py`
 285 * commit any changed files

 287 ### Upgrading to 0.15
 288 
 289 Starting with this version, Versioneer is configured with a `[versioneer]`
 290 section in your `setup.cfg` file. Earlier versions required the `setup.py` to
 291 set attributes on the `versioneer` module immediately after import. The new
 292 version will refuse to run (raising an exception during import) until you
 293 have provided the necessary `setup.cfg` section.
 294 
 295 In addition, the Versioneer package provides an executable named
 296 `versioneer`, and the installation process is driven by running `versioneer
 297 install`. In 0.14 and earlier, the executable was named
 298 `versioneer-installer` and was run without an argument.
 299 
 300 ### Upgrading to 0.14
 301 
 302 0.14 changes the format of the version string. 0.13 and earlier used
 303 hyphen-separated strings like "0.11-2-g1076c97-dirty". 0.14 and beyond use a
 304 plus-separated "local version" section strings, with dot-separated
 305 components, like "0.11+2.g1076c97". PEP440-strict tools did not like the old
 306 format, but should be ok with the new one.
 307 
 308 ### Upgrading from 0.11 to 0.12
 309 
 310 Nothing special.
 311 
 312 ### Upgrading from 0.10 to 0.11
 313 
 314 You must add a `versioneer.VCS = "git"` to your `setup.py` before re-running
 315 `setup.py setup_versioneer`. This will enable the use of additional
 316 version-control systems (SVN, etc) in the future.

 318 ## Future Directions
 319 
 320 This tool is designed to make it easily extended to other version-control
 321 systems: all VCS-specific components are in separate directories like
 322 src/git/ . The top-level `versioneer.py` script is assembled from these
 323 components by running make-versioneer.py . In the future, make-versioneer.py
 324 will take a VCS name as an argument, and will construct a version of
 325 `versioneer.py` that is specific to the given VCS. It might also take the
 326 configuration arguments that are currently provided manually during
 327 installation by editing setup.py . Alternatively, it might go the other
 328 direction and include code from all supported VCS systems, reducing the
 329 number of intermediate scripts.

 356 def get_root():

1442 def get_cmdclass():
1443     if "versioneer" in sys.modules:
1444         del sys.modules["versioneer"]
1445         # this fixes the "python setup.py develop" case (also 'install' and
1446         # 'easy_install .'), in which subdependencies of the main project are
1447         # built (using setup.py bdist_egg) in the same python process. Assume
1448         # a main project A and a dependency B, which use different versions
1449         # of Versioneer. A's setup.py imports A's Versioneer, leaving it in
1450         # sys.modules by the time B's setup.py is executed, causing B to run
1451         # with the wrong versioneer. Setuptools wraps the sub-dep builds in a
1452         # sandbox that restores sys.modules to it's pre-build state, so the
1453         # parent is protected against the child's "import versioneer". By
1454         # removing ourselves from sys.modules here, before the child build
1455         # happens, we protect the child from the parent's versioneer too.
1456         # Also see https://github.com/warner/python-versioneer/issues/52
1457 
1458     cmds = {}

1461     from distutils.core import Command

1464         description = "report generated version string"
1465         user_options = []
1466         boolean_options = []

1469             pass
1470 
1471         def finalize_options(self):
1472             pass

1475             vers = get_versions(verbose=True)
1476             print("Version: %s" % vers["version"])
1477             print(" full-revisionid: %s" % vers.get("full-revisionid"))
1478             print(" dirty: %s" % vers.get("dirty"))
1479             if vers["error"]:
1480                 print(" error: %s" % vers["error"])

1483 
1484     # we override "build_py" in both distutils and setuptools
1485     #
1486     # most invocation pathways end up running build_py:
1487     #  distutils/build -> build_py
1488     #  distutils/install -> distutils/build ->..
1489     #  setuptools/bdist_wheel -> distutils/install ->..
1490     #  setuptools/bdist_egg -> distutils/install_lib -> build_py
1491     #  setuptools/install -> bdist_egg ->..
1492     #  setuptools/develop -> ?

1502             # now locate _version.py in the new build/ directory and replace
1503             # it with an updated value
1504             if cfg.versionfile_build:
1505                 target_versionfile = os.path.join(self.build_lib, cfg.versionfile_build)
1506                 print("UPDATING %s" % target_versionfile)
1507                 write_to_version_file(target_versionfile, versions)
1508

1512         from cx_Freeze.dist import build_exe as _build_exe

1549             versions = get_versions()

1551             # unless we update this, the command will keep using the old
1552             # version
1553             self.distribution.metadata.version = versions["version"]
1554             return _sdist.run(self)
1555 
1556         def make_release_tree(self, base_dir, files):
1557             root = get_root()
