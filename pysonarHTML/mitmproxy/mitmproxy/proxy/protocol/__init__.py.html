   1 """
   2 In mitmproxy, protocols are implemented as a set of layers, which are composed
   3 on top each other. The first layer is usually the proxy mode, e.g. transparent
   4 proxy or normal HTTP proxy. Next, various protocol layers are stacked on top of
   5 each other - imagine WebSocket on top of an HTTP Upgrade request. An actual
   6 mitmproxy connection may look as follows (outermost layer first):
   7 
   8     Transparent HTTP proxy, no TLS:
   9       - TransparentProxy
  10       - Http1Layer
  11       - HttpLayer
  12 
  13     Regular proxy, CONNECT request with WebSocket over SSL:
  14       - ReverseProxy
  15       - Http1Layer
  16       - HttpLayer
  17       - TLSLayer
  18       - WebsocketLayer (or TCPLayer)
  19 
  20 Every layer acts as a read-only context for its inner layers (see
  21 :py:class:`Layer`). To communicate with an outer layer, a layer can use
  22 functions provided in the context. The next layer is always determined by a
  23 call to :py:meth:`.next_layer() <mitmproxy.proxy.RootContext.next_layer>`,
  24 which is provided by the root context.
  25 
  26 Another subtle design goal of this architecture is that upstream connections
  27 should be established as late as possible; this makes server replay without any
  28 outgoing connections possible.
  29 """
