  15     DOCUMENTATION = r'''
  16 options:
  17   mode:
  18     description:
  19     - The permissions the resulting file or directory should have.
  20     - For those used to I(/usr/bin/chmod) remember that modes are actually octal numbers.
  21       You must either add a leading zero so that Ansible's YAML parser knows it is an octal number
  22       (like C(0644) or C(01777)) or quote it (like C('644') or C('1777')) so Ansible receives
  23       a string and can do its own conversion from string into number.
  24     - Giving Ansible a number without following one of these rules will end up with a decimal
  25       number which will have unexpected results.
  26     - As of Ansible 1.8, the mode may be specified as a symbolic mode (for example, C(u+rwx) or
  27       C(u=rw,g=r,o=r)).
  28     - As of Ansible 2.6, the mode may also be the special string C(preserve).
  29     - When set to C(preserve) the file will be given the same permissions as the source file.
  30     type: raw
  31   owner:
  32     description:
  33     - Name of the user that should own the file/directory, as would be fed to I(chown).
  34     type: str
  35   group:
  36     description:
  37     - Name of the group that should own the file/directory, as would be fed to I(chown).
  38     type: str
  39   seuser:
  40     description:
  41     - The user part of the SELinux file context.
  42     - By default it uses the C(system) policy, where applicable.
  43     - When set to C(_default), it will use the C(user) portion of the policy if available.
  44     type: str
  45   serole:
  46     description:
  47     - The role part of the SELinux file context.
  48     - When set to C(_default), it will use the C(role) portion of the policy if available.
  49     type: str
  50   setype:
  51     description:
  52     - The type part of the SELinux file context.
  53     - When set to C(_default), it will use the C(type) portion of the policy if available.
  54     type: str
  55   selevel:
  56     description:
  57     - The level part of the SELinux file context.
  58     - This is the MLS/MCS attribute, sometimes known as the C(range).
  59     - When set to C(_default), it will use the C(level) portion of the policy if available.
  60     type: str
  61   unsafe_writes:
  62     description:
  63     - Influence when to use atomic operation to prevent data corruption or inconsistent reads from the target file.
  64     - By default this module uses atomic operations to prevent data corruption or inconsistent reads from the target files,
  65       but sometimes systems are configured or just broken in ways that prevent this. One example is docker mounted files,
  66       which cannot be updated atomically from inside the container and can only be written in an unsafe manner.
  67     - This option allows Ansible to fall back to unsafe methods of updating files when atomic operations fail
  68       (however, it doesn't force Ansible to perform unsafe writes).
  69     - IMPORTANT! Unsafe writes are subject to race conditions and can lead to data corruption.
  70     type: bool
  71     default: no
  72     version_added: '2.2'
  73   attributes:
  74     description:
  75     - The attributes the resulting file or directory should have.
  76     - To get supported flags look at the man page for I(chattr) on the target system.
  77     - This string should contain the attributes in the same order as the one displayed by I(lsattr).
  78     - The C(=) operator is assumed as default, otherwise C(+) or C(-) operators need to be included in the string.
  79     type: str
  80     aliases: [ attr ]
  81     version_added: '2.3'
  82 '''
