   7 # The feature flag system provides a way to turn on or off features of Sentry.
   8 #
   9 # Registering a new feature:
  10 #
  11 # - Determine what scope your feature falls under. By convention we have a
  12 #   organizations and project scope which map to the OrganizationFeature and
  13 #   ProjectFeature feature objects. Scoping will provide the feature with
  14 #   context.
  15 #
  16 #   Organization and Project scoped features will automatically be added into
  17 #   the Organization and Project serialized representations.
  18 #
  19 #   Additional feature contexts can be found under the features.base module,
  20 #   but you will typically deal with the organization or project.
  21 #
  22 #   NOTE: There is no currently established convention for features which do not
  23 #         fall under these scopes. Use your best judgment for these.
  24 #
  25 # - Set a default for your features.
  26 #
  27 #   Feature defaults are configured in the sentry.conf.server.SENTRY_FEATURES
  28 #   module variable. This is the DEFAULT value for a feature, the default may be
  29 #   overridden by the logic in the exposed feature.manager.FeatureManager
  30 #   instance. See the ``has`` method here for a detailed understanding of how
  31 #   the default values is overridden.
  32 #
  33 # - Use your feature.
  34 #
  35 #   You can check if a feature is enabled using the following call:
  36 #
  37 #   >>> features.has('organization:my-feature', organization, actor=user)
  38 #
  39 #   NOTE: The second parameter is used to provide the feature context, again
  40 #         organization and project are the most common, but it is possible that
  41 #         other Feature objects may require more arguments.
  42 #
  43 #   NOTE: The actor kwarg should be passed when it's expected that the handler
  44 #         needs context of the user.
  45 #
  46 #   NOTE: Features that require Snuba to function, add to the
  47 #         `requires_snuba` tuple.
  48
